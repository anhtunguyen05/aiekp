# Phase 17: VSCode Extension Release & Product Verification

## 1. Context & Motivation
Following the successful integration of Security, Authentication, and Multi-Tenancy (Phase 16), the AIEKP platform now requires authentication across all client endpoints (CLI, Web Dashboard, and VSCode). Currently, the CLI has been released to PyPI, but the VSCode extension has not yet been published to the Microsoft Marketplace.

Before publishing the extension, it is critical to perform end-to-end verification of the entire product ecosystem to ensure that the new auth flows work seamlessly across all components and do not break the core developer experience.

## 2. Goals
1.  **System Verification**: Thoroughly test all AIEKP components (CLI, API, Dashboard, VSCode) to verify system integrity post-Phase 16.
2.  **VSCode Integration Update**: Ensure the VSCode extension supports the new token-based authentication mechanism.
3.  **Marketplace Release**: Package (`.vsix`) and publish the VSCode extension to the Microsoft Marketplace.
4.  **Documentation Update**: Polish end-user onboarding documentation, particularly regarding authentication, to prepare for public use.

## 3. Scope of Work

### 3.1. End-to-End Product Verification
*   **Authentication Flow Verification**:
    *   Verify CLI login/register commands persist tokens correctly.
    *   Verify Web Dashboard login flow and session persistence.
    *   Verify VSCode extension settings can accept and store the `API_URL` and `Auth Token` (or integrate a login command).
*   **Knowledge Engine Integrity**:
    *   Scan a repository using the CLI (with auth) and verify data ingestion (Neo4j & Qdrant) is properly isolated by `tenant_id`.
*   **Reasoning Engine**:
    *   Test chat capabilities in the Dashboard and VSCode to ensure context is correctly fetched and constrained to the user's tenant.

### 3.2. VSCode Extension Updates
*   **Configuration**: Add configuration settings in `package.json` for `aiekp.token` (or similar) to allow users to input their JWT token obtained from the CLI/Dashboard.
*   **API Client**: Update the VSCode extension's HTTP client to attach the `Authorization: Bearer <token>` header to all requests sent to the local/remote AIEKP server.
*   **UI Polish**: Ensure proper error handling if a request fails due to `401 Unauthorized`, prompting the user to provide a token.

### 3.3. Marketplace Packaging & Publishing
*   **Asset Preparation**: Ensure `README.md`, `CHANGELOG.md`, and the extension icon are polished and ready for the public.
*   **Packaging**: Run `vsce package` to ensure the `.vsix` file builds successfully without bundling unnecessary files (`.vscodeignore`).
*   **Publishing**: Use `vsce publish` (or manual upload) using the provided Publisher ID and Personal Access Token (PAT).

### 3.4. Documentation Polish
*   **Onboarding Guide**: Update the main `README.md` to guide new users through:
    1.  Starting the backend (Docker Compose).
    2.  Installing the CLI via PyPI.
    3.  Registering/Logging in.
    4.  Scanning their first repository.
    5.  Installing and configuring the VSCode extension.

## 4. Success Criteria
*   [ ] A repository can be scanned, queried, and explored without any authentication errors across the CLI, Web, and VSCode.
*   [ ] The VSCode extension gracefully handles authentication (using a token setting).
*   [ ] The VSCode extension is successfully published to the Microsoft Marketplace and can be installed via the Extensions view.
*   [ ] The root `README.md` clearly explains the setup and authentication process.

## 5. Out of Scope
*   Implementing a complex OAuth2 web-based login flow directly inside the VSCode extension (users can copy-paste the token generated by the CLI/Dashboard for this MVP release).
*   Automated CI/CD for VSCode publishing (manual publish via `vsce` is acceptable for the alpha release).
