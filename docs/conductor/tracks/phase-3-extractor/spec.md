# Phase 3: Metadata Extractor Specification

## 1. Goal
Process the Abstract Syntax Trees (ASTs) generated in Phase 2 to extract higher-level structural data and business context metadata. This phase filters out the "noise" from the raw AST and keeps only the essential information (Classes, Functions, Docstrings) needed for the Knowledge Graph.

## 2. Prerequisites
* Phase 2 Parser Engine is completed and capable of outputting normalized `AiekpAstNode` structures.
* Python version `^3.11` is configured via `uv`.

## 3. Technical Requirements

### 3.1 Architecture
*   **Strategy Pattern**: Define a common interface for language-specific extractors.
*   **Decoupled Package**: Extraction logic resides in a new workspace package `packages/metadata_extractor`, which depends on `parser_core` but acts independently.
*   **Pydantic Models**: Extracted metadata is strongly typed using Pydantic models to ensure data integrity before being passed to Phase 4 (Knowledge Graph).

### 3.2 Metadata Extractor Package (`packages/metadata-extractor`)
*   Initialize as a new `uv` library.
*   **Metadata Models**: Define output models such as:
    *   `FunctionMetadata` (name, docstring, parameters, return type, start/end lines)
    *   `ClassMetadata` (name, docstring, methods -> list of `FunctionMetadata`, start/end lines)
    *   `FileMetadata` (file path, language, classes, standalone functions)
*   **Interface**: Define `LanguageExtractor` (ABC) with a method like `extract(ast_root: AiekpAstNode) -> FileMetadata`.
*   **Extractor Factory**: Implement a registry/factory pattern to instantiate the correct extractor based on the file extension/language.

### 3.3 Python Extractor Plugin (`plugins/python-extractor`)
*   Initialize as a new `uv` library in the workspace.
*   Dependencies: `aiekp-metadata-extractor`, `aiekp-parser-core`.
*   Implement `PythonExtractor` adhering to the `LanguageExtractor` interface.
*   Logic: Traverse the `AiekpAstNode` tree recursively.
    *   Identify `class_definition` nodes and extract to `ClassMetadata`.
    *   Identify `function_definition` nodes and extract to `FunctionMetadata`.
    *   Extract docstrings (usually `expression_statement` -> `string` immediately following a definition) and map them accordingly.

### 3.4 Integration
*   Ensure that the AST generated by `ParserFactory` (Phase 2) can be directly consumed by the `ExtractorFactory` (Phase 3).
*   Create a simple unified pipeline test script: `Source Code -> Scanner -> Parser -> Extractor -> JSON Output`.

## 4. Definition of Done (DoD)
- [ ] `packages/metadata-extractor` is created with well-defined Pydantic output models.
- [ ] `plugins/python-extractor` is created and successfully extracts Classes, Functions, and Docstrings from a `.py` file's AST.
- [ ] Integration test script proves the flow from raw source code to Pydantic Metadata models works.
- [ ] Appropriate unit tests are written for the extractor core and the python plugin with 100% pass rate.
